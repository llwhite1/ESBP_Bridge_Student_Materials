## What you are learning today

- Focus: Debug/test design.
- Main evidence to keep: expected result, actual result, test case, likely cause.
- Why it matters: Use expected-versus-actual evidence and protective tests to diagnose behavior.

## Before class / during class / after class

- Before class: Write one expected result before looking at the actual result.
- During class: Compare expected and actual without jumping to a fix.
- After class: Write one test that would catch the same problem again.

## If you get stuck

If stuck, fill the expected/actual table before proposing a repair.

Try to name the kind of evidence you need before asking for help: output, state, type, branch decision, test, loop pass, list state, or support plan.

## How this connects to ENGR 102

This turns debugging into evidence work rather than trial-and-error.
